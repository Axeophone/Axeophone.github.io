01 — Overview

Project Overview

PaceJam is a native iOS application engineered to dynamically synchronize music with a runner's cadence. The goal was to translate the app's concept into a comprehensive brand identity, build a fully functional iOS app, and validate it through real user testing.

The team consisted of four members: I served as Lead Developer alongside two designers (Mel and Kylie) and a drone cinematographer (Chris Gerlach) who contributed to the marketing trailer. My responsibilities spanned system architecture, backend and frontend development, Git workflow management, database design, and shepherding the application from initial sketches to a shipping product.

My Role: Lead Developer — responsible for system architecture, Spotify API integration, database design and implementation (SQLite/FMDB), version control infrastructure, and overall technical leadership from concept through delivery.

02 — Problem

The Problem

Runners today have access to great tools in isolation — apps like Strava and Nike Run Club collect performance data, and streaming services like Spotify provide music. But no integrated solution bridges these two domains in a meaningful way.

Musical accompaniment significantly influences the running experience: the right tempo can push a runner through a wall, and the wrong one can kill momentum. Yet the intersection between workout data and music selection remained completely unaddressed by the market.

The team identified that the real opportunity wasn't a technology problem — it was an integration problem. The gap between running metrics and music enjoyment was the market gap worth filling.

"While applications like Strava and Nike Run Club facilitate performance data collection, and streaming services provide music access, no integrated solution existed bridging these domains." — PaceJam Case Study
Strava Performance tracking — no music integration
Spotify Music streaming — no cadence awareness
PaceJam The bridge that didn't exist

03 — Solution

The Solution

The team resolved several critical early decisions — platform, music source, and pacing direction — before settling on the core architecture. The result was dual-mode functionality designed for two distinct runner mindsets.

Responsive Mode

Music adapts in real time to the runner's current cadence. As you speed up or slow down, PaceJam selects tracks from your Spotify library that match your BPM — keeping the music in sync with your body, not the other way around.

✓ For: Free runners / casual runs
Target Mode

Music sets the pace. The runner chooses a BPM target — or structures a workout with intervals — and PaceJam selects tracks that guide them toward that goal. The music becomes a pacer, not just a soundtrack.

✓ For: Structured training / interval workouts

Key Architecture Decisions

iOS Native over React Native

Apple announced "Liquid Glass," a new fluid glass interface system incompatible with React Native. Going native in Swift was necessary to build for modern iOS aesthetics — even though it required learning unfamiliar syntax.

✓ Chosen: Swift / Native iOS
User's Spotify Library

Instead of a curated music bank, PaceJam integrates directly with the user's own Spotify library. User testing confirmed this was the most valued feature — runners want their music, not ours.

✓ Chosen: Personal Spotify Library
Local SQLite over Cloud DB

Initially caching data as JSON files proved limited. When we needed persistent workout history and user data, local SQLite via FMDB was chosen over cloud-hosted MySQL to keep overhead and cost low.

✓ Chosen: SQLite / FMDB

04 — Brand & Design

Brand Identity & UI Design

Designers Mel and Kylie led the brand and visual design work. Their competitive audit of Strava, Nike Run Club, and Spotify revealed a landscape of dark, data-heavy interfaces — an opportunity to do something different.

Color & Typography

The team pivoted toward an approachable aesthetic with electric purples and neon greens — colors designed to stimulate energy and mirror the sensation of a great run. Typography: Owners Narrow for headings (bold, impactful, no screen clutter) and Owners for body text (legible at high intensity).

Mascot: Tempo

Early logo discussions produced "Tempo," the app mascot. I sketched the initial concepts; Kylie rendered and refined them in Adobe Illustrator, blending geometric minimalism with cartoon-influenced fluidity. Kylie later animated Tempo in After Effects — looping running sequences demonstrating both app modes during onboarding.

Design System

Mel and Kylie built a component library in Figma as the single source of truth. Early prototypes had information density problems — too many data points triggered cognitive overload mid-run. Through iteration, the interface evolved toward a "clean and bold" aesthetic: white space, high-visibility typography, and large touch targets sized for thumb use on the move.

05 — Development

Application Development

Version Control & Team Workflow

I gave the team a Git crash course and set up our GitHub repository. Git's branching architecture let us run parallel workstreams without stepping on each other: I developed backend logic while Mel and Kylie refined UI components on separate branches — preventing overwrites and keeping the codebase stable throughout.

Spotify API Integration

I registered a Spotify developer application and implemented OAuth 2.0 authentication. Initial setup succeeded — but we quickly discovered that Spotify had deprecated the API endpoint exposing track tempo data roughly one month before we started development. This forced months of searching for alternatives.

  • ReccoBeats — Reddit-recommended, completely nonfunctional. Python testing confirmed systematic 400 errors on every request, including verbatim documentation examples.
  • Rapid API — Nokia's API syndication platform, providing genre, mood, BPM, and Spotify track IDs. Free tier was capped at 3 calls/day. I purchased a $10/month subscription for 5,000+ monthly calls. Within a week, the first functional app build emerged.

Database Architecture

By January 2026, JSON file caching had hit its limits. The feature roadmap — workout history, data analysis, persistent user data — required a real database. After evaluating options, I chose local SQLite over cloud-hosted MySQL to minimize overhead and cost.

The team ran a whiteboard session to map tables, attributes, and relationships. I built the schema in MySQL Workbench, then converted datatypes to SQLite format using DB Browser. The result: PaceJam.DB.

Writing database code in Swift was particularly hard — it required knowledge approaching Objective-C. I found FMDB, a Cocoa wrapper around SQLite that exposed object-oriented classes for database management. This dramatically accelerated development. Over the following months, I replaced all existing feature logic with database-integrated implementations:

  • Complete Spotify library caching (songs & playlists)
  • Workout history tracking
  • Persistent login logic
  • Intelligent song selection algorithms
  • CRUD functions for every database table

Claude Code Integration

What worked: Claude rapidly read existing codebases and added features efficiently. Styling for features not covered in the Figma prototype came out production-quality on first attempts — largely because the designers had laid such a strong foundational system. Kylie used Claude to implement the UI, feeding it the Figma prototype and specific backend integration instructions to achieve all minimum, target, and reach goals.

What didn't: Claude deleted the .gitignore configuration file. Most feature additions introduced one or two breaking changes through context loss, compiler conflicts, or overwritten logic. Claude occasionally disobeyed explicit instructions — sometimes creating more work than it saved. AI assistance amplifies good foundations, but requires constant oversight.

06 — User Testing

Usability Testing & Iteration

The team conducted moderated usability testing with six Generation Z recreational runners. I facilitated interviews and questioning; Mel handled note-taking and recruitment; Kylie analyzed data and documented findings.

Results at a Glance

100% Task completion rate on core setup flows
82.1 System Usability Scale score
#1 Most valued feature: personal Spotify library over stock music

Key Iterations from Testing

1
Workflow Restructuring

Users expected to choose their workout structure before setting pace — not after. We split these into separate sequential screens, aligning the flow with how runners actually think about their workouts.

2
Terminology Clarification

"Increase/Decrease" buttons were misread as immediate pace controls. Renaming them to Negative Splits / Positive Splits resolved the confusion. "Next Lap" became Next Interval to eliminate unintended GPS tracking associations.

3
Cognitive Load Reduction

The run screen had too much information competing for attention. We redesigned the visual hierarchy — grouping interval statistics so runners could scan them in milliseconds without disrupting their stride.

07 — Results

Results & Next Steps

The team successfully transformed an ambitious concept into a functional, data-driven iOS application — from a whiteboard idea to a shipped product with real user validation. User research confirmed the core value proposition while surfacing specific improvements that made the app measurably easier to use.

The promotional trailer, shot and edited by the team, launched alongside the app at the capstone showcase — bringing the full PaceJam brand to life through video as well as software.

What's Next

  • Real-world field testing — Taking PaceJam on live runs to fine-tune the music selection algorithms under actual running conditions, where pace fluctuates unpredictably
  • Algorithm refinement — Ensuring music motivation stays consistent across the full arc of a run, not just at setup
  • Expanded mascot integration — Bringing Tempo more fully into the run experience, not just onboarding

Key Learnings:

  • Terminology matters as much as functionality — small label changes resolved major user confusion
  • AI-assisted development requires strong human oversight; it amplifies a good foundation but introduces new failure modes
  • API deprecations can derail a project; building flexibility into your data layer early is essential
  • Mascot and brand personality meaningfully reduce onboarding friction

08 — Reflection

Reflection

What Worked Well

The Git branching structure kept four people from ever blocking each other. The database whiteboard session upfront saved weeks of rework. And starting with a strong Figma design system made Claude Code integration genuinely useful rather than chaotic.

Challenges

Spotify's API deprecation mid-project was the biggest threat to delivery — it cost months of rerouting. Writing database logic in Swift approached Objective-C territory. And managing AI-assisted development required more human review than anticipated.

Next Time

I'd audit third-party API stability before building core architecture around it. I'd also establish AI usage guidelines for the team earlier — clear rules about what Claude should and shouldn't touch would have prevented several regression cycles.
